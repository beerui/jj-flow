# 可交付的项目级 AI 实践工程

最近我越来越感觉，AI 编程最容易让人产生错觉的地方，不是它写得慢，而是它写得太快。

它可以很快生成一段代码，很快补完一个页面，也很快根据几句话写出一个看起来完整的功能。但真正进入项目开发后，我发现问题并没有因为“生成速度变快”而消失，反而换了一种形式出现：测试再仔细也会漏 bug，review 再认真也会疲劳，AI 写出来的代码量越来越大，但质量、边界、风格和验证并不稳定。

所以我现在更关心的不是“怎么让 AI 帮我写更多代码”，而是另一个问题：

> 怎么把 AI 变成一个在真实项目里可以稳定交付的工程能力？

这也是我想讨论的主题：可交付的项目级 AI 实践工程。

## 这场分享应该让大家带走什么

如果只讲“我用 AI 做了什么”，这场分享的价值不够。

我希望听完的人至少带走 5 件东西：

```text
1. 一个判断：AI 开发的问题不是会不会写，而是能不能交付。
2. 一条路径：任务 → 证据 → 计划 → 实现 → 验证 → 线上反馈 → 经验沉淀。
3. 一套方法：用真实任务、YApi、代码、测试、ARMS/SLS 约束 AI。
4. 一组示例：每个案例都有可跑命令、预期产物和失败兜底。
5. 一个可复制骨架：.workflow/daji-ai-delivery 里的自有工作流 overlay。
```

不同听众能获得的东西也不一样。

| 听众 | 最应该带走的收获 |
| --- | --- |
| 前端开发 | 怎么让 AI 按项目边界写代码，而不是乱猜接口和组件 |
| Tech Lead | 怎么把 AI 从个人效率工具变成团队交付流程 |
| 测试 / QA | 怎么把验证、UAT、线上反馈放进 AI 工作流 |
| 后端 / 架构 | 怎么把接口契约、权限边界、调度验收说清楚 |
| 管理者 | 怎么判断 AI 交付不是“看起来完成”，而是有证据闭环 |

所以整场分享要围绕一个问题展开：

> AI 写出来的东西，怎么证明它真的能交付？

## 一、问题不是 AI 不会写，而是它缺少项目级约束

很多人刚开始用 AI 写代码时，体验会非常兴奋。

给它一个需求，它能写。  
给它一个报错，它能修。  
给它一个组件，它能补。  
给它一个页面，它能生成。

但用得越多，我越发现一个问题：AI 的能力很强，但它默认并不知道你的项目边界。

它不知道这个接口字段是不是来自真实后端。  
它不知道这个页面已有类似组件可以复用。  
它不知道这个功能只允许主站展示，不能影响其它租户。  
它不知道某个阶段只是前端验收包，不代表后端调度能力已经实现。  
它也不知道一个 milestone 里产物齐了，不等于验收真的通过。

这就像学生写作文。

你给他一个题目，他可能当场写出一篇看起来不错的文章。但如果他只是复制粘贴了一次答案，下次换个题目，他还是不会写。真正有价值的不是这一次作文，而是他是否理解了审题方式、结构方法、表达边界和评分标准。

AI 写代码也是一样。

一次 prompt 得到的代码，只是答案。  
能被复用、验证、迁移、复盘的流程，才是能力。

## 二、老师和学生：AI 模型只是学生

我现在更愿意用“老师和学生”来理解 AI 开发。

学生是谁？  
是 Claude、Codex、Gemini、DeepSeek 这些模型。

老师是谁？  
不是某个更强的模型，而是我们给 AI 准备的项目知识、接口文档、编码规则、测试标准、线上反馈和工作流。

如果老师没讲清楚，学生就只能靠猜。  
这个学生猜对了，不代表下个学生也能猜对。  
这次生成对了，不代表下次还能稳定生成对。

所以我真正想做的，不是找到一个“永远最强的模型”，而是建立一套“换哪个模型来，都知道该怎么做”的项目教学体系。

就像写作文一样。

AI 帮你写了一篇作文，你复制粘贴了，这次可能拿到了结果。  
但下次没有 AI、没有这段 prompt、题目换一下，你还是不会写。

真正有价值的不是那篇作文，而是你沉淀下来的写作方法。

代码也是一样。  
一次生成的代码只是结果。  
能被复用的规则、流程和验证方式，才是能力。

## 三、我在 `seo-daji-web` 遇到的真实问题

这个想法不是凭空来的，是我在做 `seo-daji-web` 时慢慢感受到的。

比如做 AI 获客功能时，如果直接让 AI 开始写代码，它很容易看起来很积极，但实际风险很多。

第一个问题是：需求来源不稳定。  
有些文档路径写的是一个版本，正文标题又像另一个版本。如果 AI 不先搞清楚真实需求，就会基于错误上下文继续往下写。

第二个问题是：前后端边界容易混。  
前端仓库里能做的是页面、状态、按钮逻辑、接口调用、验收用例。但后端调度任务有没有实现，不在这个仓库里。AI 如果没被提醒，很容易把“我写了验收方案”说成“这个能力已经完成”。

第三个问题是：阶段编号容易误解。  
我执行 `maestro-analyze 1`、`quality-review 2` 这类流程时，表面上只是数字 1、2，但真实要做的是对应的 phase。这个对应关系不能靠猜，要回到 `.workflow/state.json` 和 `phase_slugs` 里确认。

第四个问题是：验证不能只看起来很完整。  
分析文件、计划文件、执行文件、review 文件都有，不代表功能就真的过了。如果 UAT、后端人工验收、verification 还有 pending，就不能随便给 `PASS`。

第五个问题是：测试也不是越大越好。  
在这个项目里，默认跑一个很大的检查不一定最有效。更靠谱的做法是围绕本次改动跑目标化测试、目标化 lint，再检查 diff。验证应该回答“这次改动有没有真的影响正确的地方”，而不是制造一个看起来很正式的流程。

这些问题让我意识到：AI 开发真正难的地方，不是让它写代码，而是让它在真实项目边界里稳定交付。

## 四、结合 AI 进行开发：最佳实践的猜想

我的猜想是：AI 开发的最佳实践，不应该从 prompt 开始，而应该从项目交付链路开始。

一个真实项目里，开发不是孤立发生的。

需求可能来自禅道。  
接口可能来自 YApi。  
已有写法藏在代码里。  
线上问题来自 ARMS。  
最后还要回到 Bug 状态、工时、周报和复盘。

所以 AI 不应该只接收一句“帮我实现这个功能”。

更好的方式应该是：

先让 AI 明确任务来源。  
再让 AI 读取真实接口。  
再让 AI 学习项目里已有的同类写法。  
然后生成计划。  
再开始写代码。  
写完之后跑验证。  
如果线上出问题，再把问题反向沉淀成规则。

这样 AI 才不是一个临时帮手，而是进入了真实研发流程。

## 五、真实项目下的工作流实践

这里我希望保留我之前在 Linux.do 里记录的“问题 / 解决方案”写法。它不是理论，而是我真实遇到的问题和逐步摸出来的处理方式。

### 1. UI 设计图还原不准确

问题：AI 还原设计图不稳定，即使用了 MCP，也可能只能做到“看起来差不多”。

解决方案：不要只让 AI 看设计稿就直接写代码。可以先让模型结合截图和 MCP 还原，再加入自我验证：截图对比、细节检查、还原度说明。如果要进一步稳定，就把 UI 验证写进工作流，让 AI 生成验证规则、验证步骤和审查文档。

核心不是“这次还原得像不像”，而是下次遇到 UI 还原需求时，有一套固定验证流程。

### 2. 本地环境问题会干扰 AI 开发

问题：Windows 下容易遇到乱码、命令不准确、浏览器插件不稳定等问题。

解决方案：先把基础环境稳定住。比如 Windows 命令行乱码可以先执行：

```bash
chcp 65001
```

如果是 Claude Code 这类更适合类 Unix 环境的工具，可以放到 WSL 里使用。也有人选择直接切到 Mac 环境。这里的重点是：不要把环境问题误判成 AI 能力问题。环境不稳定，工作流再好也会被打断。

### 3. 在已有项目里修 Bug，直接对话容易反复

问题：已有项目中修 Bug，如果只靠单次对话，可能要来回很多轮，甚至改了几轮没改好，还引出新问题，也没有留下可复用记录。

解决方案：使用工作流，让 AI 明确哪些问题已经确认、哪些方案已经尝试、提测范围是什么。

单一问题修复可以这样走：

```text
maestro debug
  梳理问题链路，找到现象、复现方式和正确修复方向

maestro plan
  针对已确认 issue 生成修复计划

maestro execute
  按计划拆解并执行修改

maestro quality-review
  对代码进行审查

maestro quality-test
  做必要验证

maestro learn
  把有价值的经验沉淀下来，下次类似问题直接参考
```

简单记法：

```text
现象不明：debug
根因明确但影响大：issue_plan
计划确认后：execute
改完后：quality-review / test
有可复用经验：learn / spec add
```

### 4. 简单问题也需要留痕

问题：简单改动如果只让 AI 直接改，很快就完成了，但后面很难知道它理解了什么、改了哪里、边界是什么。

解决方案：简单问题可以走 quick，但 quick 也要有留痕。

例如：

```bash
maestro-quick MarketOverviewSection.vue:239 这里所有的推荐都放开，不再限制 5 个
```

即使是这种小改动，也应该至少留下三类信息：

```text
需求理解 + 涉及模块 + 修改边界
任务规划
任务状态管理
```

这能避免“改是改了，但不知道为什么这么改”的问题。

### 5. 单会话计划、执行、审查容易自我证明

问题：同一个会话里让 AI 计划、执行、审查，它很容易自我辩解、自我证明。比如多语言场景漏改了，当前对话里的 review 也可能看不出来。

解决方案：组队。根据职责拆成不同的子 Agent 或不同角色：一个负责实现，一个负责 review，一个负责测试，一个负责找遗漏。

这背后的原则很简单：不要让同一个“学生”既写答案又给自己判卷。

### 6. 不希望 AI 继续使用某种写法，要沉淀成规则

问题：项目里不需要某些写法，但 AI 会被老代码影响，继续在新需求里使用。比如某个项目不需要额外使用 `localePath`，AI 却因为历史代码继续加。

解决方案：把“以后不要这么写”的偏好写进项目记忆或 spec。

例如：

```bash
/spec-add coding "避免无必要 localePath" "在 project-web 中，除非现有链路明确需要 locale 前缀路由，不要新增 const localePath = useLocalePath() 或无意义包 localePath(...)" --keywords localePath,i18n,routing --description "避免无 localePath 包装"
```

不同类型的经验应该放到不同分类里：

```text
coding：以后代码怎么写、怎么不写
debug：某个问题为什么出错、怎么排查
arch：架构边界和长期设计决策
review：代码评审时按什么标准看
test：测试怎么跑、怎么补
```

这样 AI 下次不是靠当前 prompt 临时记住，而是能从项目规则里读到。

### 7. 修 Bug 要能留痕、计划、执行和提交

问题：修一个 Bug 不是“改到不报错”就结束了。真正麻烦的是根因有没有确认、修复有没有验证、同类问题有没有扫完、最后有没有留下提交和记录。

解决方案：可以用更长周期的 debug 工作流，比如 `odyssey-debug`。

它适合处理这种目标：

```text
根因必须确认到底
修复必须验证通过
同类问题要扩散扫描
发现的新问题要一起处理
最后要有清晰记录
```

这类流程看起来比 quick 慢，但适合高风险 Bug。因为它追求的不是快，而是不要留下尾巴。

### 8. 一个需求已经有 PRD 和接口设计，应该怎么执行？

问题：需求已经有 PRD 和接口设计，但如果直接让 AI 开始写代码，很容易跳过理解和拆解。

解决方案：走完整需求落地流程。

1. 规划、规范蓝图

先把 PRD 和接口文档变成完整规格包，内容包括需求边界、业务流程、接口依赖、数据流、架构说明、模块拆分、Epics 和开发阶段拆分。

```bash
/maestro-blueprint "读取 @docs/v17.1/接口文档.md prd.md，生成完整规格包：需求边界、业务流程、接口依赖、数据流、架构说明、模块拆分、Epics 和开发阶段拆分" -y
```

2. 目标分解，路线分析

根据 blueprint、analyze 或 brainstorm 的产出，继续拆 milestone、phase、优先级和依赖关系。

```bash
/maestro-roadmap "基于 @docs/v17.1/接口文档.md prd.md 和 blueprint 结果，拆分 milestone、phase、优先级、依赖关系" -y
```

3. 正式推进 Phase 1

```bash
/maestro-analyze 1
```

如果希望自动执行，可以增加 `-y`。

4. 生成可执行、可追溯的计划

```bash
/maestro-plan 1
```

5. 确认计划没问题后执行

```bash
/maestro-execute 1
```

6. Phase 1 执行完后走质量检查

```bash
/quality-review 1
/quality-auto-test 1
/quality-test 1
```

之后再进入 Phase 2。

7. 整个 milestone 完成后，再审计并标记完成

```bash
/maestro-milestone-audit
/maestro-milestone-complete
```

### 9. 完成一个功能后，怎么让 AI 下次照着做？

问题：我完成了一项功能、一个页面，希望 AI 下次做类似需求时完全按照这套规则来做，要怎么办？

解决方案：不要只留下代码，要把这次实现沉淀成可复用规则。

这一步我还想继续补完整，但方向已经很明确：做完一个代表性功能后，需要让 AI 回头整理这次实现。

要整理的不是“改了哪些文件”这么简单，而是：

```text
这个功能解决了什么问题
涉及哪些页面、组件、接口和数据结构
复用了项目里的哪些既有写法
哪些地方是业务边界，不能随便改
哪些验证结果能证明它真的完成了
下次遇到类似需求时，应该先读哪些文件、按什么步骤做
```

最后把这些内容沉淀到 spec、knowhow 或 workflow 里。这样下次 AI 再做类似页面时，不是重新猜一遍，而是先学习这次已经验证过的规则。

### 10. 线上异常和流量暴涨，怎么用 ARMS / SLS 反推代码问题？

问题：测试再仔细，也挡不住线上真实流量。比如 `seo-daji-web` 的 `goods-detail` 页面出现访问暴涨、实例压力变大时，如果只看代码，很容易变成“我猜哪里慢”；如果只看日志，又很难知道应该改哪段代码。

真实工作对话：`019f2159-90f6-7290-aec2-033ff0800bc7`。

解决方案：用 `arms-fix` 或 ARMS / SLS 链路先拿真实证据，再回到代码里找放大点。

这次分析不是上来就改代码，而是先确认几件事：

```text
先确认精确 app.id，不能只看 env 里的 PID 前缀
再看目标时间窗和前后相邻时间窗
再看 referrer、initial_load、distinct URL 和 resource 数量
再回到代码里看 SSR、hydration、接口请求是否被放大
最后才决定限流、缓存、降级和压测方案
```

这次得到的关键信息很具体：

```text
09:30-10:00，goods-detail 有 370 次 view
约 191 个 session
约 211 个不同详情页 URL
大多数 referrer 为空
top 商品页最高只有 7 次 view
09:00-09:30 已经有 429 次 view
10:00-10:30 还有 637 次 view
```

这说明它不像是某一个商品爆火，更像是一批详情页被持续直接访问。再结合代码看，`goods-detail` 冷 SSR 会拉详情接口，客户端水合后还会触发 fresh 详情、推荐、物流、字典、价格计算等请求，流量会被进一步放大。

另一个关键点是：不能看到 `nitro.maxConcurrency: 80` 就以为已经有限流。实际排查时发现，在当前 `nitropack@2.12.4` 的运行面里，不能把它当成已经生效的 Node server 保护阀。

所以这类问题的工作流应该是：

```text
ARMS / SLS 拉真实数据
  ↓
确认 app.id、时间窗、访问形态
  ↓
回到代码分析 SSR 和客户端请求链路
  ↓
确认配置是否真的生效
  ↓
落地限流、缓存、503 保护或削峰方案
  ↓
压测和观测验证
  ↓
沉淀成线上问题处理规则
```

这里 `arms-fix` 的价值不是“帮我修一个报错”这么简单，而是把线上反馈变成可执行的工程闭环：补分析、定位根因、隔离分支、改源码、验证、提交、归档。

### 11. 已经有计划方案时，接口参数和返回值不能靠猜

问题：AI 很容易根据 PRD、字段名、已有代码习惯去猜接口。看起来很快，但真实项目里接口路径、请求参数、返回结构只要猜错一点，后面代码、类型、测试都会跟着偏。

真实工作对话：`019f1b4c-00a6-7080-89cd-b2aae4f4ac74`。

解决方案：在 plan 阶段就用 `yapi` 拉真实接口文档，把接口契约变成任务的必读资料，而不是等写代码时临时猜。

这次 AI 获客需求里，PRD 里已经列了多个 YApi 地址。正确做法不是让 AI 看一眼表格就开始写，而是先把接口拉下来：

```bash
/yapi http://yapi.sdpjw.com/project/1437/interface/api/100591 --base-path /gateway/api/business-service-web
```

或者用底层命令：

```bash
yapi-tool fetch "http://yapi.sdpjw.com/project/1437/interface/api/100591" --format md --base-path "/gateway/api/business-service-web"
```

真实流程是这样的：

```text
先根据 PRD 和 roadmap 确认当前 phase
  ↓
用 yapi-tool 拉取真实接口 Markdown
  ↓
把接口文档放进 plan 会话目录
  ↓
生成任务时写入 read_first
  ↓
把接口字段、返回结构、guard 一起写进计划
  ↓
执行阶段按真实接口契约写代码和测试
```

这次还发现一个很有价值的点：YApi 请求体里出现了 `currentUserId`、`currentUserName`，但项目既有规范是前端默认不显式传用户和租户字段，而是交给 token / header 透传。

如果 AI 没有这个 guard，很可能直接把字段写进请求体，看起来“符合接口文档”，但实际违背项目习惯。

所以正确处理不是马上删，也不是马上加，而是在计划里写清楚：

```text
YApi 中存在 currentUserId / currentUserName
但前端默认不主动传用户身份字段
执行阶段需要确认后端是否强依赖
未确认前，不让 AI 自己猜
```

这就是 `yapi` 融入工作流的价值：它不只是拉接口文档，而是让 AI 在计划阶段就面对真实接口、真实字段和真实边界。计划不是“我要做什么”的作文，而是“我拿什么证据做、哪些地方不能猜”的执行说明。

### 12. 放到完整研发链路里

如果把上面的案例合起来，真实项目下的工作流大概是这样：

```text
禅道任务 / Bug
  ↓
YApi 接口文档
  ↓
项目已有代码学习
  ↓
AI 拆解计划
  ↓
AI 编码实现
  ↓
测试 / lint / diff / review
  ↓
ARMS 线上反馈
  ↓
禅道关闭 / 工时记录 / 周报
  ↓
沉淀为 spec / knowhow / workflow
```

这里每个工具都有自己的位置。

`sd-zentao-cli` 解决任务入口和交付出口。  
它让 AI 面对的不是一句模糊描述，而是真实任务、真实 Bug、真实状态。

`yapi` 解决接口契约。  
它让 AI 不再猜字段、猜路径、猜响应结构，而是根据真实 API 文档开发。

`arms-fix` 解决线上反馈。  
测试再仔细也会漏问题，ARMS 就是线上真实世界给我们的反馈。它可以把异常变成一条修复链路：分析、改代码、验证、commit、归档。

`maestro-flow` 可以作为底座。  
我想二开它，不是为了多几个命令，而是把这些真实经验固化成流程。

## 六、第三方工作流存在的问题

第三方工作流最大的问题是：它们通常是通用的，但项目不是通用的。

通用工作流可以告诉 AI：

先分析。  
再计划。  
再执行。  
再 review。  
再测试。

这当然有价值，但还不够。

因为它不知道我的项目里：

哪些目录是真实需求来源。  
哪些接口必须从 YApi 读取。  
哪些功能只允许主站展示。  
哪些验证命令更可信。  
哪些结果不能提前说成 `PASS`。  
哪些线上异常需要走 ARMS。  
哪些任务最后要回到禅道。

所以第三方工作流适合作为底座，但不能直接当成最终答案。

真正能交付的工作流，一定要长进项目里。  
它要知道项目的边界，知道团队的习惯，知道历史踩过的坑，也知道哪些事情不能乱说、不能乱改、不能跳过验证。

## 七、属于自己的交付方法：工作流

所以我最终想做的，不只是写一篇文章，也不是简单收集几个 AI 工具。

我想沉淀一套属于自己的交付方法。

这套方法的核心不是“AI 帮我写得更快”，而是：

```text
让 AI 知道任务从哪里来。
让 AI 知道接口以什么为准。
让 AI 知道项目里同类功能怎么写。
让 AI 知道哪些边界不能越过。
让 AI 知道写完以后怎么验证。
让 AI 知道线上出错以后怎么复盘。
让 AI 每次做完，都能给下一次留下经验。
```

这就是我理解的项目级 AI 实践工程。

它不是一次 prompt。  
不是某个模型的高光表现。  
也不是单纯提升代码生成速度。

它是一套能持续变稳的工作流。

我希望二开 `maestro-flow`，把 `sd-zentao-cli`、`yapi`、`arms-fix` 这些真实开发工具接进去，最后形成一条自己的项目级 AI 交付链路：

```text
任务进入 → 上下文准备 → 学习项目写法 → 拆解计划 → 实现 → 验证 → 线上反馈 → 复盘沉淀
```

如果这条链路能跑通，那 AI 就不只是“帮我写代码”，而是开始变成项目里的交付能力。

## 八、演讲版闭环：这次分享要讲清楚什么

如果把这篇内容拿出去分享，我不希望它变成“我用了哪些 AI 工具”的工具介绍。

真正要讲清楚的是一个闭环：

```text
AI 生成代码很快
  ↓
但真实项目需要边界、证据和验证
  ↓
所以要把 AI 放进研发链路
  ↓
用真实任务、真实接口、真实代码、真实线上反馈约束它
  ↓
每次交付后再沉淀成下一次可复用的规则
```

可以按这个结构讲：

### 1. 开场：AI 最大的错觉不是慢，而是太快

AI 能很快写代码，但快不等于可交付。  
测试会漏，review 会累，模型会猜，项目边界会被冲破。

这一段要把问题说透：我们不是缺一个更会写代码的 AI，而是缺一套能把 AI 管进项目里的方法。

### 2. 核心比喻：老师和学生

模型是学生，项目规则、接口文档、测试标准和线上反馈才是老师。

老师讲不清楚，学生就靠猜。  
老师讲得清楚，换一个学生也能写出接近的答案。

这就是项目级 AI 实践的核心：不要追某一次生成结果，要建设一套能反复教学的系统。

### 3. 真实案例：`seo-daji-web` 为什么不能直接让 AI 写

这里讲 AI 获客、YApi、主站边界、phase 编号、后端调度边界、验证不能乱给 `PASS`。

重点不是“这个功能多复杂”，而是说明：真实项目里的信息来源很多，AI 必须知道谁是事实来源。

### 4. 工作流方法：先拿证据，再写代码

把研发过程拆成这几类证据：

```text
任务事实：禅道 / PRD / Bug
接口事实：YApi
代码事实：已有组件、已有工具函数、已有测试
验证事实：node test / eslint / diff / review
线上事实：ARMS / SLS
沉淀事实：spec / knowhow / workflow
```

这时候 AI 的角色就变了：它不是凭空生成，而是在证据之间做连接。

### 5. 现场演示：三个最有价值的片段

分享时不用把整条链路全跑完，容易超时。更适合演示三个片段：

```text
YApi 演示：接口不能猜，要拉真实契约
Maestro 演示：需求不能散聊，要变成 phase / plan / task
ARMS 演示：线上问题不能靠感觉，要用 SLS 数据反推代码瓶颈
```

完整命令和演示稿我单独放到 [Demo Kit](/other/project-ai-delivery-flow-demo-kit) 里。

### 6. 收束：二开工作流的目标

最后要落到一个可讨论的问题：

```text
如果我们把这些经验二开进 maestro-flow，
能不能让 AI 每次都按真实项目的交付规则工作？
```

这会比单纯展示“AI 又写了一个功能”更有价值。因为它讨论的是工程体系，而不是一次生成表演。

## 九、`maestro-flow` 二开计划和实施方案

我想二开 `catlog22/maestro-flow`，但第一步不应该是直接大改核心代码。

更短、更稳的路径是：先把它当成工作流底座，用项目覆盖层、工具适配器和演示配方把我的真实研发链路接进去。只有当覆盖层无法表达时，再改核心编排。

### 1. 二开目标

这次二开的目标不是“多几个命令”，而是让工作流满足 4 个要求：

```text
能接真实任务：禅道、PRD、Bug、线上异常
能接真实接口：YApi 拉取的请求参数和返回值
能接真实反馈：ARMS / SLS 的异常、访问量和根因线索
能沉淀经验：spec、knowhow、review rule、test rule
```

换句话说，它要从“通用 AI 工作流”变成“我的项目级交付工作流”。

### 2. 兼容现有实例的方式

现在已有的实例不要推倒重来，而是映射到同一条交付链路里：

| 实例 | 当前能力 | 接入后的工作流位置 | 关键产物 |
| --- | --- | --- | --- |
| AI 获客 | PRD + YApi + phase 执行 | 需求进入、接口契约、计划执行 | blueprint、roadmap、plan、task、verification |
| YApi 工具 | 拉真实接口 Markdown | 接口事实来源 | yapi markdown、contract guard、read_first |
| ARMS / SLS | 线上异常和流量分析 | 线上反馈入口 | findings、root cause、resolution、限流方案 |
| 禅道工具 | 任务和工时出口 | 任务入口与交付闭环 | issue、worklog、weekly report |
| spec / knowhow | 经验沉淀 | 下一次执行的老师 | coding rule、debug note、review rule |

### 3. 推荐架构

```text
maestro-flow core
  负责编排：analyze / plan / execute / review / test / learn

project overlay
  负责项目规则：目录边界、验证命令、主站限制、不能乱给 PASS

tool adapters
  负责真实外部输入：sd-zentao-cli、yapi-tool、arms-inspector

demo recipes
  负责可演示：每个案例都有命令、输入、输出、验收点
```

这套结构的好处是：通用流程仍然通用，项目经验长在 overlay 和 adapter 里。以后换项目，不需要重写整个工作流，只需要换项目规则和工具适配。

### 4. 最小可用版本

第一版只做 3 条黄金路径：

```text
功能交付路径：
禅道 / PRD → YApi → analyze → plan → execute → review → test → learn

线上修复路径：
ARMS / SLS → findings → fix 分支 → 修复 → 验证 → resolution → learn

经验沉淀路径：
已完成实现 → 提取规则 → 写入 spec / knowhow → 下次执行前自动加载
```

只要这 3 条跑通，就已经不是工具清单，而是可交付体系。

### 5. 实施阶段

#### Phase 1：整理协议

先定义所有工具交给 maestro 的统一输入输出。

```json
{
  "source": "yapi|arms|zentao|manual",
  "artifact_type": "contract|finding|issue|rule|verification",
  "path": ".workflow/...",
  "summary": "给 AI 读的一句话摘要",
  "guards": ["不能猜字段", "不能越过后端边界"],
  "evidence": ["可验证文件或命令"]
}
```

这一步的目标是让外部工具产物都能被 `maestro-plan`、`maestro-execute`、`quality-review` 读懂。

#### Phase 2：接入工具适配器

先接 3 个最有价值的适配器：

```text
yapi adapter：输入 YApi URL，输出接口 Markdown + contract guard
arms adapter：输入 ARMS task / URL，输出 findings + 修复边界
zentao adapter：输入任务编号，输出任务描述 + 验收状态 + 工时出口
```

适配器不负责“替 AI 思考”，只负责把真实资料整理成稳定上下文。

#### Phase 3：做项目 overlay

把 `seo-daji-web` 的真实规则写进去：

```text
phase 数字必须从 .workflow/state.json 的 phase_slugs 反查
接口字段必须优先看 YApi
前端默认不显式传 currentUserId / currentUserName
不能把后端验收包说成后端能力已完成
scoped 验证优先 node --test、目标化 eslint、git diff --check
正式人读产物默认中文
```

这一步是“老师”的核心内容。

#### Phase 4：补演示配方

每条路径都要有可演示脚本：

```text
演示目标
前置条件
命令
预期输出
失败时怎么兜底
观众应该看到什么价值
```

这部分已经放到 [Demo Kit](/other/project-ai-delivery-flow-demo-kit)。

#### Phase 5：做验收标准

二开不是写完命令就结束，要有验收标准：

```text
同一个需求换模型执行，计划结构应基本一致
接口字段不能靠猜，必须能追到 YApi 产物
线上问题必须能追到 ARMS / SLS 证据
每次完成后必须留下 verification 和 learn 产物
报告里 pending / guard / fail 不能被写成 PASS
```

能满足这些标准，才算真正进入项目级 AI 交付。

### 6. 当前已经落地的自有 workflow kit

为了避免“二开计划”只停留在文档里，我把最小可用骨架放到了仓库里：

```text

├── examples/

    ├── feature-delivery.md

```

它现在已经能做 3 件事：

## 十一、其它：两个可以接入工作流的工具

最后再补充两个我已经在做的工具，它们可以作为这套工作流里的基础能力。

第一个是 ARMS 巡检工具：[@shendu-sdt/arms-inspector](https://www.npmjs.com/package/@shendu-sdt/arms-inspector)。

它解决的是线上异常发现和归因问题。真实项目里，测试再仔细也会有漏掉的情况，线上 ARMS 数据就是最后一层反馈。这个工具的价值不是简单地“看报错”，而是把线上异常变成可以处理的任务：发现异常、聚合指纹、分析根因、进入修复流程，再把经验沉淀回工作流。

第二个是 YApi 接口拉取工具：[@shendu-sdt/yapi-tool](https://www.npmjs.com/package/@shendu-sdt/yapi-tool)。

它解决的是接口上下文问题。AI 写接口相关代码时，最怕凭空猜字段、路径和响应结构。这个工具可以把 YApi 里的真实接口信息拉取出来，整理成 AI 更容易使用的上下文，让后续的计划、实现和测试都基于真实接口契约。

一个偏线上反馈，一个偏接口输入。  
它们一个负责告诉 AI “线上真实发生了什么”，一个负责告诉 AI “接口真实长什么样”。  
这两类信息接进来以后，AI 才更像是在真实项目里工作，而不是只在一段 prompt 里想象。

## 十二、可以继续抛出来的讨论主题

这篇分享最后还可以抛两个更大的问题，让大家一起讨论。

### 1. 团队合作怎么做：Claude Code 的智能体协作给了什么启发？

这里要注意一点：我不想把“Claude Code 团队内部到底怎么工作”讲成未经证实的内幕。

更稳妥的说法是：从 Claude Code 的公开文档和研究分析里，已经能看到一种团队协作方向。

Claude Code 有 subagent、agent teams、worktree 等能力。subagent 更像“单会话里的专门工种”，适合把探索、review、测试、调研这些任务放到独立上下文里做；agent teams 更像“多个 Claude Code 会话组成的小队”，有 lead、teammates、共享 task list、mailbox，可以让不同智能体并行研究、互相沟通、最后由 lead 汇总；worktree 则解决并行修改时的文件冲突问题。

可以参考这几个公开资料：

- [Claude Code subagents](https://code.claude.com/docs/en/sub-agents)
- [Claude Code agent teams](https://code.claude.com/docs/en/agent-teams)
- [Claude Code worktrees](https://code.claude.com/docs/en/worktrees)
- [Dive into Claude Code](https://arxiv.org/abs/2604.14228)

这给我的启发是：未来团队协作不只是“一个人 + 一个 AI”。

更可能变成：

```text
人类负责人
  定目标、定边界、定验收标准

主智能体 / lead agent
  拆任务、分配、汇总、判断是否继续

专业智能体
  分别做探索、实现、测试、review、性能、安全、文档

项目工作流
  管状态、管证据、管权限、管验收、管沉淀
```

真正值得讨论的是：人类团队应该怎么分配“人”和“智能体”的职责？

例如：

```text
哪些任务适合单智能体顺序执行？
哪些任务适合多个智能体并行？
哪些任务必须有人类审批？
哪些结果可以让智能体互相 review？
哪些结论必须落到文件、测试或线上数据里？
```

我的判断是：智能体协作不是把团队里的每个人都替掉，而是把很多“等待、搜索、验证、对照、重复检查”的工作交出去。人类更像 reviewer、owner 和产品判断者。

所以我二开 `maestro-flow` 时，也不应该只做一个线性命令链。它应该能支持角色协作：

```text
explorer：只读探索，找文件、找模式、找风险
planner：把证据变成计划和任务
executor：按计划实现
reviewer：只看风险、回归和边界
tester：跑目标化验证，补测试
archivist：把经验沉淀成 spec / knowhow
```

这套角色不一定都要同时跑。关键是让每个智能体有清楚边界，不要让同一个模型既写答案，又给自己判卷。

### 2. Harness 会不会越来越不重要？

我觉得这个问题很适合讨论，但要先把 `harness` 说清楚。

这里的 `harness`，可以理解成包在大模型外面的“工程外壳”：prompt 模板、任务拆解器、工具调用胶水、多 agent 编排器、retry 机制、检查器、状态机、权限控制、评测脚本等等。

早期模型能力弱，所以 harness 很重要。  
模型不会规划，就用外层流程帮它规划。  
模型不会稳定调用工具，就用外层框架帮它固定步骤。  
模型上下文短，就用外层系统帮它检索、压缩、总结。  
模型容易忘边界，就用外层规则不断提醒。

但随着模型能力变强，很多“为了弥补模型不聪明”的 harness 会变得没那么重要。

比如：

```text
以前要写很复杂的 prompt 链，模型才会拆任务
以后模型自己就能拆得不错

以前要强行规定每一步怎么想
以后只要给目标、证据和边界，模型就能自己推进

以前要很多小 agent 才能补足能力
以后一个强模型可能就能完成大部分顺序任务
```

所以如果 `harness` 的价值只是“把弱模型包装得像强模型”，那它确实会越来越不重要。

但这不代表工程系统不重要。

真正会留下来的，不是重型 harness，而是交付护栏：

```text
证据：真实 PRD、YApi、ARMS、代码、测试结果
权限：哪些能改，哪些不能改，什么时候需要人批准
状态：当前做到哪一步，哪个 phase，哪个 task
隔离：worktree、fix 分支、工具权限、子智能体边界
验收：测试、review、UAT、线上观测
沉淀：spec、knowhow、复盘、失败经验
```

也就是说，harness 的位置会变化。

它不再是“替模型思考的大脑”，而是“让模型在项目里安全交付的工程制度”。

这也会影响我二开 `maestro-flow` 的方向：不要做一个越来越厚、越来越复杂、处处指挥模型怎么想的框架，而是做一个更薄、更清晰的交付层。

```text
少写死模型怎么推理
多提供真实证据
少做复杂 prompt 魔法
多做状态、权限、验收和复盘
少追求框架控制感
多追求结果可验证、可回滚、可复用
```

一句话总结：

> 大模型越强，补智商的 harness 越不重要；但管边界、管证据、管交付的 workflow 会越来越重要。

这也正好回到这篇分享的主题：AI 实践工程的重点，不是把模型绑进一个复杂框架，而是让它进入真实项目的交付闭环。
