# 命令参考

## 命令格式

Codex 内使用 `$jj`：

```text
$jj <命令> <自然语言输入>
```

`<自然语言输入>` 可以包含需求、资料路径、接口链接、设计图、Codex 线程、验收标准和必须由用户拍板的决策。`jj-flow` 不要求把这些内容拆成固定 CLI 参数。

安装 skill 时使用仓库提供的安装命令：

```bash
jj install-skill [--project | --target <dir>] [--force] [--dry-run] [--json]
```

## 通用输入

所有命令都接受下面这些信息：

- `目标`：要完成什么。
- `资料`：PRD、接口文档、设计图、截图、日志、diff、Codex 线程链接。
- `范围`：本次做什么，不做什么。
- `关键决策`：必须由人确认的业务取舍。
- `验收`：什么结果算完成。

写法示例：

```text
$jj delivery
目标：完成 AI 获客列表、详情和精修。
资料：PRD 在 docs/v17.1，接口来自 YApi，设计图来自 MasterGo。
范围：本期不做导出。
验收：页面还原设计，接口字段真实，测试和 review 通过。
```

## `$jj delivery`

### 用途

少参数端到端交付。适合一个需求需要经历分析、计划、实现、审查、测试、UI 精修和沉淀。

### 输入参数

- `需求`：必填，一句话或多段都可以。
- `资料`：可选但推荐，包括 PRD、接口文档、设计图、Codex 线程。
- `关键决策`：可选，写清用户已拍板的取舍。
- `验收标准`：可选，缺失时由模型根据资料先推导，再保持待确认。

### 使用方案

当需求较完整时，直接把资料一起给 `$jj delivery`。当资料分散时，只给线索也可以，模型会先自动发现项目上下文和 `.workflow`。

```text
$jj delivery 按 PRD、接口文档和设计图完成 AI 获客页面，本期不做导出
```

### 输出期望

- 自动整理 context package。
- 生成 Maestro 调用链。
- 标出 guard 状态。
- 只在阻塞决策时追问用户。
- 完成后保留测试、review、UAT 或待人工验证状态。

## `$jj validate`

### 用途

项目管理者自检。适合检查 `jj-flow` 当前状态、文档与代码一致性、recipe/guard/测试覆盖、workflow 状态和下一步升级建议。

### 输入参数

- `自检目标`：可选，不写时默认检查当前项目状态。
- `范围`：可选，例如只看文档、recipe、guard、测试或路线图。
- `证据`：可选，例如路线图、失败日志、用户反馈、已完成 diff。
- `升级偏好`：可选，例如优先减少用户参与次数、优先完善安装体验、优先补测试。

### 使用方案

```text
$jj validate 检查 jj-flow 当前状态，找出文档、recipe、guard、测试和路线图的漂移
```

```text
$jj validate 基于当前路线图和用户反馈，给出下一步最该实现的项目管理能力
```

### 输出期望

- 读取项目状态、`.workflow` 和路线图。
- 检查文档、recipe、guard、测试、workflow 是否一致。
- 报告 Maestro CLI 是否可用以及版本是否兼容。
- 标出 `PASS`、`PENDING`、`FAIL` 或需人工确认项。
- 生成下一步建议，并说明证据来源。

## `$jj evolve`

### 用途

项目迭代管理。适合在 `$jj validate` 之后，把自检结果、路线图、用户反馈和当前仓库状态整理成下一轮升级计划，并明确哪些问题应自动修正、哪些决策必须由用户确认。

### 输入参数

- `迭代目标`：可选，不写时默认基于当前自检和路线图推进下一步。
- `优先级`：可选，例如先补自检失败、先完善安装、先减少用户参与次数。
- `约束`：可选，例如不改 Maestro core、不引入重型编排引擎、不做外部工具直连。
- `证据`：可选，例如 `$jj validate` 输出、路线图、用户反馈、测试失败日志。

### 使用方案

```text
$jj evolve 基于当前自检结果推进下一项项目管理能力
```

```text
$jj evolve 优先减少用户参与次数，但不要重写 Maestro 的执行能力
```

### 输出期望

- 先复用 `$jj validate` 的项目自检证据。
- 生成 correction backlog，优先列出必须修正的漂移。
- 给出下一轮升级计划、验收标准和建议 Maestro 调用链。
- 明确保持 `jj-flow` 是 Maestro 上层协议，而不是新的执行引擎。

## `$jj feat`

### 用途

明确的小功能交付。适合已有需求边界和接口资料，只需要按项目模式实现。

### 输入参数

- `功能目标`：必填。
- `接口资料`：涉及接口时推荐提供。
- `页面或模块`：推荐提供路径、组件名或路由。
- `验收标准`：推荐提供。

### 使用方案

```text
$jj feat 在客户列表增加筛选项，接口字段看 YApi 链接，验收以现有列表页交互为准
```

### 输出期望

- 先确认边界和证据。
- 生成实现计划。
- 调用 `$maestro-plan`、`$yapi`、`$maestro-execute`、`$quality-review`、`$quality-test` 等链路。

## `$jj fix`

### 用途

线上问题定位与修复。适合 ARMS/SLS、报错、回归、异常行为。

### 输入参数

- `现象`：必填，用户看到的问题。
- `时间窗`：推荐提供，便于查日志。
- `错误指纹`：推荐提供，例如 ARMS、SLS、console error。
- `影响范围`：推荐提供，例如页面、租户、用户路径。

### 使用方案

```text
$jj fix 线上 goods-detail 在 09:30 到 10:00 出现 500，ARMS 指纹如下，要求定位根因并最小修复
```

### 输出期望

- 先拿真实日志或复现证据。
- 说明根因机制、引入位置和修复位置。
- 修复范围尽量窄。
- 验证复现路径和回归点。

## `$jj review`

### 用途

交付前质量审查。适合检查计划、diff、测试结果、发布风险。

### 输入参数

- `审查目标`：必填。
- `diff 或变更范围`：推荐提供。
- `验收标准`：推荐提供。
- `风险关注点`：可选，例如兼容性、性能、权限、租户范围。

### 使用方案

```text
$jj review 审查 AI 获客页面本次改动，重点看接口字段、设计还原、权限和测试缺口
```

### 输出期望

- 先列风险和缺陷。
- 每个结论追溯到文件、命令或证据。
- 没有证据的通过项保持 `PENDING`。

## `$jj knowhow`

### 用途

把真实交付过程沉淀成可复用知识。适合总结 Codex 线程、提交、问题和解决方案。

### 输入参数

- `沉淀对象`：必填，例如一次需求、一个 bug、一组 Codex 线程。
- `复用目标`：推荐提供，例如 workflow、spec、recipe、排查路径。
- `证据`：推荐提供，例如线程链接、提交、文档、测试结果。

### 使用方案

```text
$jj knowhow 总结这三个 Codex 线程的完整交付流程，沉淀成后续 delivery 模板
```

### 输出期望

- 保留问题、约束、证据、方案和复用条件。
- 区分一次性事实和可复用规则。
- 可沉淀为 knowhow、spec 或 workflow recipe。

## `$jj auto`

### 用途

自动判断意图。当前文档不推荐把 `auto` 当主入口；真实交付优先用 `$jj delivery`，明确任务优先用 `$jj fix`、`$jj review` 或 `$jj knowhow`。

### 使用方案

```text
$jj auto 帮我判断这件事应该走交付、修复、审查还是沉淀
```

### 输出期望

- 选择最合适的 recipe。
- 给出选择理由。
- 仍保持证据不足不通过的 guard 规则。

## `jj install-skill`

### 用途

把本仓库的 `skills/jj` 安装到 Codex skill 目录，让 Codex 能识别 `$jj`。这是安装命令，不是 Codex 对话里的交付命令。

### 参数

- `--project`：安装到当前项目的 `./.codex/skills/jj`。
- `--target <dir>`：安装到指定 skill 目录，默认是 `~/.codex/skills/jj`。
- `--force`：目标目录已存在时覆盖文件。
- `--dry-run`：只显示将要安装的位置，不写文件。
- `--json`：输出结构化结果，便于脚本检查。

### 使用方案

首次安装：

```bash
npx @shendu-sdt/jj-flow@beta install-skill
```

升级已有安装：

```bash
npx @shendu-sdt/jj-flow@beta install-skill --force
```

只安装到当前项目：

```bash
npx @shendu-sdt/jj-flow@beta install-skill --project
```

预览安装位置：

```bash
npx @shendu-sdt/jj-flow@beta install-skill --dry-run
```

### 输出期望

- 安装成功时，`~/.codex/skills/jj/SKILL.md` 存在。
- 目标已存在且未传 `--force` 时，命令失败并提示如何覆盖。
- `--json` 输出包含 `ok`、`status`、`source`、`target` 和 `message`。

## 选择建议

- 完整需求：用 `$jj delivery`。
- 项目自检和下一步升级：用 `$jj validate`。
- 推进项目自身迭代：用 `$jj evolve`。
- 明确新增功能：用 `$jj feat`。
- 线上问题或异常：用 `$jj fix`。
- 交付前把关：用 `$jj review`。
- 总结和沉淀：用 `$jj knowhow`。
- 不确定分类：用 `$jj auto`，但不要把它当长期主入口。
