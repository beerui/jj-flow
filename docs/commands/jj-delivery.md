# `jj-delivery` 端到端交付

`jj-delivery` 是默认的少参数交付入口。你可以只描述自然语言需求，也可以附上 PRD、YApi、设计稿、日志、截图、diff 或历史线程；Agent 负责自动发现上下文，并组织分析、计划、实现、验证、审查和必要的知识沉淀。边界明确的小功能、线上问题最小修复与交付前审查也统一走本入口（已移除 `jj-feat` / `jj-fix` / `jj-review`）。

## 适用场景

- 一个真实需求需要跨过需求理解、资料核对、编码、测试和审查多个阶段。
- 资料散落在仓库、`.workflow`、接口平台、设计稿或历史线程中，不想先手工整理成固定参数。
- 目标、接口和验收已经明确的小到中型功能开发。
- 已有线上现象、错误指纹、回归路径，需要定位根因并做最小修复。
- 页面交付还包含 UI 精修、真实接口校对或发布前质量把关。

## 何时不用

- 同源项目族的迁移从当前领头项目分析阶段转入 `$jj-same`；不要直接复制整分支到 sibling。
- 维护 `jj-flow` 自身时使用 [`jj-validate`](command-jj-validate.html) / [`jj-evolve`](command-jj-evolve.html)。
- 纯审查也走本入口：优先 `$quality-review`，不擅自扩改未点名范围。
- 知识沉淀不设独立入口；交付完成后可按需调用 Maestro 的 knowhow 相关 skill。

## 输入模板

```text
$jj-delivery
目标：<最终要交付的业务结果>
资料：<PRD、接口、设计、截图、日志、线程或文件路径>
范围：<本期包含什么>
不做范围：<明确排除什么>
验收：<功能、交互、测试、审查或发布标准>
关键决策：<已经拍板的取舍，或必须由你决定的问题>
```

Claude Code 把首行改成 `/jj-delivery`。除了目标，其余信息都可以先省略；Agent 会优先从当前项目自动发现。

## 完整示例

### 示例 1：页面、接口和设计一起交付

```text
$jj-delivery
目标：完成 AI 获客列表、详情页和最终视觉精修。
资料：PRD 在 docs/v17.1；接口来自 YApi 项目 100591；设计稿使用 MasterGo 链接；已有列表页可作为代码模式参考。
范围：列表查询、分页、详情展示、加载态、空态和错误态。
不做范围：本期不做导出、批量操作和权限体系重构。
验收：字段以 YApi 为准，交互还原设计稿，目标测试和 quality review 通过，未覆盖风险单独列出。
关键决策：分页沿用现有项目组件，不引入新的表格库。
```

### 示例 2：边界清楚的小功能

```text
$jj-delivery
目标：在账号安全页增加修改密码弹框。
资料：接口见 YApi /account/password，交互沿用当前页面的验证码弹框。
范围：只在当前页面弹出，不新增路由，不改旧登录逻辑。
验收：验证码优先，密码入口保留，目标单测通过。
```

### 示例 3：线上问题最小修复

```text
$jj-delivery
现象：09:30 到 10:00 登录页出现 TypeError，并且同一个错误提示重复弹出。
资料：ARMS 指纹为 login-1027，相关截图和请求 ID 已附在对话中。
范围：只修根因和重复提示，不调整登录交互。
验收：能解释引入点，最小修复，聚焦回归测试通过。
```

### 示例 4：交付前审查

```text
$jj-delivery
目标：审查支付状态优化是否可发布。
资料：当前分支 diff，目标测试结果，验收文档在 docs/payment-status-acceptance.md。
范围：只做风险与测试缺口审查，不主动扩改未点名功能。
验收：按严重度输出 findings，未验证项保持 PENDING。
```

## 执行过程

1. 还原原始目标、范围、不做范围和验收标准。
2. 自动读取当前项目、`.workflow`、PRD、接口、设计、日志、历史线程和相邻实现。**禁止调用 `maestro explore`**；定位使用 Read、Glob、Grep、Bash 或已批准 skill。
3. 先报告证据清单与缺口；证据不足的关键结论保持 `PENDING`。
4. 根据规模调用 Maestro/Codex 的分析、blueprint、roadmap、plan、execute、review、test 或 UI 精修能力；线上异常可选用 `$arms-fix`。
5. 按已确认边界实现或做最小修复，运行与风险匹配的聚焦验证。
6. 汇总修改、验证、残余风险和仍需人工确认的事项。
7. 若属于同源项目族且源项目验证通过，由 `$jj-same` 做 post-change discovery 和家族交付计划更新；后续项目不会自动开始。

## 输出/完成标准

- 需求目标、范围、不做范围和验收标准可追溯。
- PRD、接口、设计、现有代码和测试等证据已列清，缺口没有被猜测补齐。
- 有可执行计划，且实现没有超出已确认边界。
- 修改经过聚焦测试、必要静态检查和质量审查；无法运行的验证明确保持 `PENDING`。
- 最终报告包含修改内容、验证命令与结果、残余风险和人工确认项。
- 同源项目场景只完成当前授权项目，并形成可恢复交接，不自动修改所有 sibling。

## 常见误区

- 要求用户先整理 `--prd`、`--api`、`--design` 才能开始。资料路径可以直接写在自然语言中，也可以由 Agent 自动发现。
- 把每个需求都升级为大型 blueprint 或重构。流程应按真实复杂度选择最短链路。
- 证据不足仍给出“已通过”的结论。缺少接口、设计或运行时验证时应标记 `PENDING`。
- 通过 shell 执行 `jj-delivery`。真实入口是 Codex 的 `$jj-delivery` 或 Claude Code 的 `/jj-delivery`。
- 调用 `maestro explore`。全部流程禁止。
- 当前项目验证完成后自动进入其它同源项目。后续项目需要门禁通过，并由用户在新会话主动触发。
